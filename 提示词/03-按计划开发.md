# 按计划开发指令

## 指令内容

```
Follow plan.md for feature development: When developing new features, strictly follow the steps listed in the plan.md. Each step is arranged in order and must be completed sequentially. After completing each step, modify the plan.md file to add the word "Done" and a two-line summary of the step. This ensures a clear work log, which helps maintain transparency and track progress.
```

## 指令含义

**核心原则：有序推进，记录追踪**

这个指令要求AI严格按照plan.md文件中列出的步骤进行功能开发，确保开发过程的有序性和可追踪性。每完成一个步骤都要在plan.md中标记并总结。

## 适用场景

### 1. 复杂功能开发
- 多模块协同的大型功能
- 需要分阶段实施的项目
- 涉及多个技术栈的集成

### 2. 团队协作项目
- 多人参与的开发任务
- 需要进度同步的项目
- 要求透明度的开发过程

### 3. 长期维护项目
- 需要历史记录的系统
- 持续迭代的产品功能
- 质量要求严格的项目

## 执行规范

### plan.md文件结构

```markdown
# [项目名称] 开发计划

## 概述
[项目描述和目标]

## 实施步骤

### 阶段1：[阶段名称]

#### 步骤1：[步骤名称] - Done
- 目标：[具体目标]
- 完成情况：[完成总结第一行]
- 关键成果：[完成总结第二行]

#### 步骤2：[步骤名称]
- 目标：[具体目标]
- 输入：[输入要求]
- 输出：[预期输出]
- 验证：[验证方法]

### 阶段2：[阶段名称]
...
```

### 标记规范

#### 完成标记格式
```markdown
#### 步骤X：[步骤名称] - Done
[原有内容保持不变]

**完成总结：**
- [总结第一行：主要完成内容]
- [总结第二行：关键成果或发现]
```

#### 总结要求
1. **第一行**：简述主要完成的工作内容
2. **第二行**：说明关键成果、发现或影响
3. **简洁明了**：每行不超过50个字符
4. **客观准确**：基于实际完成情况

## 工作流程

### 1. 开始前检查
```markdown
检查清单：
- [ ] plan.md文件存在且结构完整
- [ ] 当前步骤明确定义
- [ ] 前置步骤已完成
- [ ] 所需资源已准备
```

### 2. 执行过程
```markdown
执行原则：
- 严格按照步骤顺序执行
- 不跳过任何定义的步骤
- 每个步骤完成后立即标记
- 遇到问题时记录和解决
```

### 3. 完成后更新
```markdown
更新步骤：
1. 在步骤标题后添加" - Done"
2. 添加两行完成总结
3. 检查下一步骤的准备情况
4. 提交plan.md的更新
```

## 异常处理

### 步骤阻塞处理
```markdown
处理流程：
1. 记录阻塞原因
2. 评估影响范围
3. 制定解决方案
4. 更新计划时间线
5. 在plan.md中记录变更
```

### 计划调整
```markdown
调整原则：
- 重大调整需要重新评估整体计划
- 小幅调整在当前步骤中说明
- 所有调整都要在plan.md中体现
- 保持调整的可追溯性
```

1. **严格顺序**：不能跳过步骤，必须按顺序执行
2. **及时更新**：每完成一步立即更新plan.md
3. **准确记录**：总结要客观反映实际完成情况
4. **保持同步**：确保plan.md始终是最新状态
5. **质量优先**：不为了进度而降低质量标准
